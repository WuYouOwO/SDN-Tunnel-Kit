跨境 L3 Overlay 与 BGP 引导的双向策略出口

在两台互不可直连、地址族错配的 NAT 主机(CMCC-GD 家宽 ⇄ HGC 香港)之间,基于 ipip+FOU over VMess 建立点到点三层隧道,用 BIRD/BGP 统一路由,并验证按前缀双向引导互联网出口。
全部目标达成 · 主链路清理后健康
日期 2026-07-02 节点 Home-SDN-PoC ⇄ HGC-NAT-114514 数据面 ipip/FOU · VMess-AEAD/TCP 控制面 BIRD 2.x · eBGP 65002/65001

执行摘要

目标与结论一览。

PoC 2 在 PoC 1(打通隧道本身)基础上,验证该 Overlay 可作为一张可编程的 SDN 底座:控制面用 BGP 宣告任意前缀,数据面即可把该前缀的互联网流量引导到指定节点出口。两次镜像实验均成功并经 conntrack 定量坐实:

  • 方向 A — 从 CMCC 宣告 218.241.97.42/32,使 HGC 去该目标的流量经 CMCC 出口。往返 ~71 ms。通过
  • 方向 B — 从 HGC 宣告 5.161.7.195/32,使 CMCC 去该目标的流量经 HGC 出口。往返 ~267 ms。通过

关键工程发现:该跨境路径上的 SDN 中继会破坏 REALITY 的 TLS 握手,但放行裸 TCP;据此将传输从 VLESS+REALITY 切换为 VMess-AEAD/TCP(无 TLS 握手、有机密性),隧道随即稳定。两条测试路由在验证后已按要求全部撤销,基础隧道与 BGP 会话保持在线。

2可达性约束 — 为何必须经中继

两端无法直连,唯一通路是运营商的 SDN 中继。

  • HGC:私网 IPv4、NAT 后、完全无 IPv6;只能靠中继/端口转发被动被访问。
  • CMCC:私网 IPv4(家宽 NAT)、只有全局 IPv6 出口,无公网 IPv4。
  • 因此 CMCC(IPv6-only)与 HGC(IPv4-only)地址族相反、彼此不可直达。
唯一可行通路 = 一条 IPv6→IPv4 的 SDN 中继:
CMCC ──IPv6──▶ [2401:b60:2a:f2f4:…]:14061 ──▶ 118.141.151.167:21122 ──(NAT-VPS 转发 21122→21122)──▶ HGC 10.102.0.110:21122
中继标识:SDN 中继 HGC-01 · 香港IPV6-CMIN2 ×0.5 · TCP

由此确定角色:CMCC = bridge(向中继拨号),HGC = portal(在 21122 监听),由 Xray reverse 打通 NAT 穿透。

4控制面:BIRD / BGP

在隧道链路 10.0.100.0/30 上跑 eBGP,自动互换各自 LAN 前缀。

会话与路由(稳态)

  • CMCC portal ↔ HGC bridge:均 Established
  • CMCC 学到 172.19.1.0/24 via 10.0.100.1 [AS65001]
  • HGC 学到 172.19.2.0/24 via 10.0.100.2 [AS65002]
  • 跨 LAN 双向 ping 0 丢包(~28–30 ms)

宣告机制

本地 protocol direct 引入 dummy0 前缀;protocol kernel {export all} 把 BGP 学到的路由写进内核;策略引流则临时加 protocol static 宣告 /32,并用 kernel 导出过滤避免本机 blackhole(见 §6)。

5关键发现 — 中继破坏 REALITY 的 TLS 握手

最初 VLESS+REALITY 完全无法握手;逐层排障后定位为中继对 TLS 的干扰。

排查项结果
服务端日志REALITY: handshake did not complete successfully / failed to read client hello
公私钥 / UUID / SNI / shortId逐一核对一致(公钥 cRo7B208… ← 私钥 oGnD…)排除
时钟偏差< 7 秒 排除
回落目标可达性HGC→www.microsoft.com:443 可达 排除
PROXY-protocol 头纯 VLESS 可通 → 中继未加头 排除
纯 VLESS(明文)测试立即打通(~27 ms,0 丢包) → 中继放行裸 TCP、仅破坏 TLS 握手 根因
结论与对策:该跨境中继会打断 REALITY 依赖的完整 TLS/uTLS 握手(疑似首包 SNI 嗅探/分段重组),但对不含 TLS 的裸 TCP 透明。故传输改用 VMess-AEAD/TCP —— 无 TLS 握手、payload 加密、可穿透中继。任何依赖干净 TLS 的方案(REALITY / WS+TLS / gRPC+TLS)在此路径上均有失败风险。

6PoC 2 实验 — BGP 引导的双向策略出口

同一条隧道,靠 BGP 宣告方向即可决定"从哪一端出网"。方法三要素:①BGP 宣告 /32;②出口端 ip_forward=1;③出口端对该目的做 NAT(MASQUERADE)。宣告端用 kernel 导出过滤,避免把该 /32 当 blackhole 写进本机。

9结论与后续

PoC 2 证明:即便两端都在 NAT 后、出口地址族相反、仅有一条会破坏 TLS 的中继,仍可构建一条稳定的加密 L3 Overlay,并以 BGP 作为唯一控制面实现按前缀、双向的策略出口选择。该底座可直接演进为多站点 SD-WAN / 智能分流。

建议后续

  • 规范化策略出口:把"某端作为一批网段的出口"做成模板(BGP 前缀集 + 转发 + NAT 一键落盘持久)。
  • 传输加固:评估 Xray VLESS-Encryption(无 TLS 的原生加密 + 前向保密)替代 VMess,兼顾穿透与安全。
  • 健壮性:中继 IPv6 前端疑似轮换 → 配 DDNS/多前端;补 BGP BFD 加快收敛。
  • 运维护栏:HGC 的 portal 配置在 s-ui confdir 内,面板改动可能覆盖手改,需纳入变更核对。

A附录 — 关键配置与验证命令

策略出口(以方向 B 为例,出口端 = HGC)

# ① BIRD:宣告 /32 给对端,并让本机按默认路由转发(不 blackhole)
protocol static test_announce { ipv4; route 5.161.7.195/32 blackhole; }
protocol kernel { ipv4 { import none;
  export filter { if net = 5.161.7.195/32 then reject; accept; }; }; }

# ② 开转发 + 对该目的做 NAT(出口网卡 eth0)
sysctl -w net.ipv4.ip_forward=1
iptables -t nat -A POSTROUTING -d 5.161.7.195/32 -o eth0 -j MASQUERADE
iptables -I FORWARD -i tun-fou -o eth0 -j ACCEPT
iptables -I FORWARD -i eth0 -o tun-fou -j ACCEPT

验证

# 对端(发起方)确认收到并从隧道走
birdc show route 5.161.7.195/32 all
ip route get 5.161.7.195         # via 10.0.100.1 dev tun-fou
ping -c5 5.161.7.195             # 经对端出口
# 出口端坐实 NAT
grep 5.161.7.195 /proc/net/nf_conntrack

撤销

# 删 bird 的 test_announce + 恢复 kernel export all → birdc configure
iptables -t nat -D POSTROUTING -d 5.161.7.195/32 -o eth0 -j MASQUERADE
sysctl -w net.ipv4.ip_forward=0   # 对端 BGP 撤回后,发起方路由自动消失
